## ★★ 이 작업의 본질 (최우선 마인드셋 — 2026-07-13 사용자 명시)
> **이건 "번역"이 아니라 "현지 맥락에 맞춰 재구성한 포스팅"이다.** 누가 EN을 더 100% 충실하게 옮기느냐가 목표가 **아니다**.
> - EN 내용을 **기반(base)**으로 삼되, **그 언어권 독자·검색 습관·문화 맥락에 맞게 포스트를 다시 구성**한다. 직역·미러링 금지.
> - 그 시장에서 **원래부터 그 언어로 쓰인 글처럼** 읽혀야 한다(제목·훅·예시·비유·문단 흐름·SERP 각도를 현지 기준으로 재저작).
> - 판단 기준: "이 문장을 EN 원문과 대조했을 때 얼마나 똑같은가"가 아니라 **"이 언어 네이티브 독자가 읽을 때 자연스럽고 이 시장 검색에 맞는가"**.
> - **단 예외 = §13(숫자·카드·확률·산수)은 절대 불변.** 사실·수치는 그대로, 그 외 표현·구성은 현지화 자유도 높게.
> 서브에이전트 번역 프롬프트·QA 프롬프트에 이 마인드셋을 항상 포함할 것.

## 0. 착수 전 준비 (매 언어 1회, 반복 프로세스)
번역 dispatch 전에 **메인이** 다음을 수행하고, 그 결과를 각 서브에이전트에 참고자료로 넘긴다.
1. **페르소나 자동부여** — 각 번역 서브에이전트에 「해당국 네이티브 번역 전문가 + 현지 씬에서 수십 년 뛴 텍사스 홀덤 전문가」 페르소나를 준다(그 사람이 직접 쓰듯 번역).
2. **현지어 사전 웹리서치** — 대상 언어로 실제 검색해 **현지에서 쓰는 포커 용어·문법·표기 관습**을 파악한다. 확인 항목:
   - 포커 핵심 용어를 현지어로 쓰는지 vs 영어 그대로인지 (예: blinds, call, raise, showdown, all-in, side pot, kicker …)
   - **숫자 표기**(천단위·소수점 구분자), 화폐 관습(€/$/현지통화), 의문/괄호 부호(전각 등)
   - 검색자가 실제 치는 롱테일(현지 SERP·PAA)
   - 존칭/문체 관습(예: de의 Du/Sie, es의 tuteo/vosotros)
   → 언어별 **용어·표기 브리프**로 정리해 서브에이전트 프롬프트에 포함(문서화: `docs/translation-terms-<lang>.md` 선택).
> 이 준비가 "기계번역 티" 방지의 핵심. ja=가타카나 관습, es=천단위 점·소수점 콤마처럼 언어마다 함정이 다르다.

## 0.5 ★표준 파이프라인 (24개 언어로 검증된 end-to-end — 신규 언어/필라 모두 이 절차)

1. **웹리서치 → 브리프**: 대상 언어로 실제 검색해 현지 포커 용어·문법·표기·문체 파악 → `docs/translation-terms-<lang>.md` 작성(§0). 이미 있으면 재사용.
2. **번역(글당 1 서브에이전트, 6병렬, Opus)**: 「현지 네이티브 번역가 + 수십 년 홀덤 전문가」 페르소나 + §6 마인드셋(재구성 포스팅) + 브리프 첨부. 구조 레퍼런스는 **`lib/posts-es/<slug>.ts`**(필드 순서·export 형태만; **텍스트 복사 금지**). RTL 언어(ar·fa·he)는 **`lib/posts-ar/`를 구조 레퍼런스로**(텍스트 복사 금지). ⚠️ **다른 현지어 파일(특히 id↔ms, ru↔uk)을 베끼지 말 것** — 언어별 새 저작.

   - `masterUpdated` = 각 EN 마스터의 `updated` 값 기록(스테일 정밀비교용, §스테일).
3. **조립**: 메인이 `lib/posts-<lang>/index.ts`에 필라 글 먼저 그룹핑(주석 `// Rules 필라 (6/6)`) + 기타. `category` 표준 필라값, `readTime` 현지 단위 통일.
4. **기계검사(하드게이트)**:
   - `npm run build` → **빌드 성공 + 페이지 수 증가 확인**(신규 로케일당 +8: home+blogindex+6글). 라우트 파일 없으면 데이터가 맞아도 **URL 404**(2026-07-13 실사고).
   - `node scripts/check-intl-links.mjs`(내부링크 하드페일) + `npm run check:stale`(필라 글 누락).
   - **숫자표기 grep**: 유럽식 언어면 영어식 `1,326`/`2.5` 잔류 0 / 영어식 언어면 반대. **비라틴 숫자**(fa 페르시아·bn 벵골·hi 데바나가리) 잔류 0 → 있으면 **라틴 일괄변환(.mjs)**.
   - FAQ 마커 `**Q. …?**`+`A.`(리터럴) 유지 확인(스키마).
5. **적대적 QA 2배치(Opus, 각 3편)**: 회의적 현지 편집장 페르소나. §13 **베스트5 재검산**·숫자표기·용어 일관성·**타언어 잔재**(영/스/한국어, ru↔uk·ar↔he 오염)·링크(/lang/blog+화이트리스트)·메타(desc≤160)·(RTL이면 우→좌 렌더) 전수 → 🔴(필수)/🟡(제안) 리포트.
6. **교정**: 메인이 `.mjs`(readFileSync→replace/split·join→writeFileSync, ⚠️Windows 파일락) 일괄 적용 → 재빌드. 임시 스크립트 삭제.
7. **커밋+push**(파일럿N) + 보고서(배포현황·§13·현지화 하이라이트·백로그). **사용자 몫=GSC 색인**(신규 언어×글수 URL 목록 제공, RTL은 렌더 육안확인 권장).

## ⓪ 신규 로케일 배선 체크리스트 (재사용·검증됨 — 빠지면 404)

② `lib/posts-<lang>/index.ts` (글 배열).

④ community translate route: `LANG_NAME`에 언어명 추가.
⑤ **★라우트 3파일 생성**(`app/tr/`가 템플릿): `app/<lang>/page.tsx`(CommunityClient) · `app/<lang>/blog/page.tsx`(IntlBlogIndex) · `app/<lang>/blog/[slug]/page.tsx`(`generateStaticParams`=`postsForLocale("<lang>")` + IntlBlogArticle).

## 저작 방식
- 대상 EN 원문(`lib/posts-en/<slug>.ts`)을 읽고, **같은 필드 구조**로 `lib/posts-<lang>/<slug>.ts`를 저작(`export const POST: Post = {...}`).
- 기존 같은 언어 글 1편을 열어 **톤·용어 관습**을 참고할 것(있으면).

## 언어·용어 규칙
- **네이티브 자연스러움 최우선** — 직역 금지. 그 언어 포커 커뮤니티가 실제 쓰는 표현·문장 리듬.
- **포커 용어 = 현지 관습.** 그 언어권에서 통용되는 포커 용어를 쓴다. **현지어로 옮기면 어색하거나 통하지 않는 용어는 영어 용어를 그대로**(예: all-in, side pot, showdown, kicker, bluff 등은 대개 영어 그대로가 자연스러움 — 언어별 판단).
- **현지화(로컬라이제이션)**: 예시·화폐·문화 레퍼런스가 그 시장에 더 자연스럽게 바뀌는 게 맞으면 **자연스러운 맥락 한정 수용**. ★단, 바꾼 것은 **전부 보고**(아래 "현지화 노트").
- **§13 수치·카드·핸드 표기는 절대 변경 금지** — 숫자·카드 표기(A♠ K♥ 등)·확률·산수는 원문 그대로. 번역이 수치를 훼손하면 치명(마스터 오류 전파와 동급).

## 슬러그·메타
- **slug 동일**(원문과 정확히 같게 — hreflang).
- **seoTitle/desc는 직역 금지** → 현지 검색자가 실제 치는 표현으로 재작성. **desc ≤ 160자**, seoTitle은 현지 SERP 잘림 고려(대략 ≤ 60 권장, 언어별 폭 감안).
- category는 원문 값 그대로 유지(표준 6필라 값).
- tags도 현지 검색어로 현지화(직역 나열 금지).

## ★내부링크 (빌드 하드체크 — 어기면 배포 차단)
- 본문·`:::readnext`·Related의 내부 블로그 링크는 **반드시 `/<lang>/blog/슬러그`** 형식.
- **"허용 슬러그 목록"에 있는 대상만 링크로 유지.** 목록에 없는 슬러그(=아직 이 언어로 번역 안 됨)로 향하는 링크는:
  - 본문: **링크 마크업만 제거하고 앵커 텍스트는 남긴다**(예: `[text](/en/blog/x)` → `text`). `thumb:` 이미지 참조도 함께 제거.
  - `:::readnext`/Related 카드: 그 항목(줄/카드) **통째 삭제**.
- 즉 이 언어에 존재하지 않는 글로는 절대 링크하지 않는다(같은 언어 404 = 빌드 실패).

## 구조 보존
- 마크다운 표, `==하이라이트==`/`==g:==`/`==r:==`, HTML `<div>`/Related 카드, 이미지 마크다운(이미지 파일은 공용 — 경로 그대로, **alt만 번역**), `:::readnext`, FAQ `**Q.**/A.` 패턴(스키마), tldr, layout 필드 — 전부 원문 구조 유지하고 **값만 번역**.
- 이미지 alt에 든 카드 표기(A♠ 등)는 그대로.

## 스테일(뒤처짐) 방지
- EN 마스터가 갱신되면 기존 번역이 조용히 옛 버전으로 남는다(스테일). `npm run check:stale`로 감지:
  같은 슬러그에서 EN `updated` > 번역본 `updated` 이면 "재번역/검토 필요"로 리포트(빌드 차단 아님).
- **정밀도 관례(권장)**: 번역 저작 시 `masterUpdated: "<그 시점 EN updated 날짜>"`를 Post에 기록하면,
  번역본 자체 수정으로 updated가 바뀌어도 마스터 대비 정확 비교가 된다(스테일 체커가 masterUpdated 우선 사용).
  신규/재번역 시 이 필드를 EN 원문의 updated 값으로 넣을 것.

## 출력 + 보고
- 파일: `lib/posts-<lang>/<slug>.ts` (Write). `import type { Post } from "../posts";` + `export const POST: Post = {...}`.
- 저작 후 **현지화 노트**를 5~10줄로 보고:

  4. seoTitle/desc 현지 SERP 각도 요약
  5. 애매했던 판단(사용자 확인 필요할 수 있는 것)
